# 明日方舟基建规划助手实现方案（V1）

## 1. 项目目标与边界

### 1.1 目标
- 输入：用户提供的干员列表截图（可多张）。
- 输出：
  1) 识别后的“用户已拥有干员清单”；
  2) 基于干员基建技能的排班建议（制造站/贸易站/发电站/控制中枢等），并给出可解释的收益估计与替代方案。
- 设计原则：
  - 不强求全局最优，但要“稳定高效、可解释、可落地”。
  - 结果允许用户手动微调。

### 1.2 范围（V1）
- 支持简体中文客户端常见分辨率截图。
- 支持核心基建设施排班：制造站、贸易站、发电站、控制中枢、会客室、训练室（可按优先级分阶段实现）。
- 支持“缺少关键干员时给次优方案”。
- 暂不做：
  - 全自动实时识别（仅离线上传截图）；
  - 多语言 OCR；
  - 复杂体力恢复动态模拟（V2 再加入更精细疲劳模型）。

---

## 2. 系统拆分与总体架构

按你的思路拆成两大模块，并补齐公共层：

1) **模块 A：截图识别（Roster Extractor）**
- 负责把干员截图解析为结构化干员列表。
- 输出统一数据结构：`OwnedOperator[]`。

2) **模块 B：排班规划（Scheduling Engine）**
- 输入用户干员列表 + 基建技能规则库。
- 输出排班方案 + 预估收益 + 备选组合。

3) **公共支撑层（Shared Infra）**
- 规则数据管理（从文档整理成可计算数据）。
- 术语映射（OCR 名称纠错、别名映射）。
- 评估器（统一的收益计算函数）。

推荐技术形态：
- 后端：Python（OCR 与算法生态成熟）。
- 前端：Web（React/Vue 任一）展示结果与手动调班。
- 数据：JSON/YAML 起步，后续可迁移 SQLite。

## 3. 模块 A：干员截图识别

## 3.1 处理流程
1. **输入校验**：检查图片清晰度、是否为干员列表界面。
2. **图像预处理**：
   - 裁边、降噪、亮度/对比度增强；
   - 透视矫正与缩放统一。
3. **卡片定位**：检测每个干员卡片区域（模板匹配或轻量检测模型）。
4. **文本识别（OCR）**：识别干员名、精英化状态、等级（V1 可先只要名字）。
5. **名称归一化**：
   - OCR 纠错（编辑距离 + 词典匹配）；
   - 别名映射到标准干员 ID。
6. **去重与置信度融合**：多张截图合并，保留最高置信度。
7. **人工确认页**：低置信度项提示用户确认。

## 3.2 建议实现
- OCR：PaddleOCR 或 Tesseract（优先 PaddleOCR 中文效果）。
- 名称匹配：
  - 维护 `operators_alias_map.json`；
  - 使用 `rapidfuzz` 做近似匹配。
- 输出结构示例：

```json
{
  "operators": [
    {
      "id": "exusiai",
      "name_cn": "能天使",
      "owned": true,
      "elite": 2,
      "level": 60,
      "confidence": 0.93
    }
  ],
  "unresolved": ["可颂?", "德狗?"],
  "source_images": ["img_001.png", "img_002.png"]
}
```

## 3.3 风险与对策
- **风险**：截图风格差异导致 OCR 漏识别。
- **对策**：
  - 增加“手动增删干员”入口；
  - 保留识别中间可视化（框选结果）便于调试；
  - 建立小规模标注集持续迭代。

## 4. 模块 B：基建排班规划

## 4.1 数据建模（核心）
先将网页文档中的基建技能结构化成规则库（建议人工校对后入库）：

- `OperatorSkillRule`
  - `operator_id`
  - `facility_type`（manufacture/trade/power/control/...）
  - `effect_type`（产能%、订单效率、心情消耗修正、联动增益等）
  - `effect_value`
  - `conditions`（与哪些干员同驻、时间段、是否精英化）
  - `tags`（例如：龙门币、作战记录、贵金属）

- `FacilityState`
  - 当前基建布局（如 252 / 243）
  - 每个设施可容纳人数
  - 班次时长（例如 12h）

- `OperatorState`
  - 当前可用状态（在岗/休息）
  - 心情值（V1 可简化为二值可用）

## 4.2 排班目标函数
定义一个可解释的评分函数：

`Score = 主产出收益 + 联动加成收益 - 疲劳惩罚 - 切换成本`

其中：
- 主产出收益：制造/贸易的单位时间收益换算；
- 联动加成收益：组合技能触发时额外分值；
- 疲劳惩罚：连续上岗过久的衰减（V1 可简化）；
- 切换成本：频繁换班的操作负担。

## 4.3 求解策略（V1 推荐）
采用“**规则候选 + 启发式搜索**”而非重型全局优化：

1. 先根据设施类型生成可行候选组合（过滤未拥有干员）。
2. 使用规则优先级（高价值组合优先）构造初始解。
3. 用局部搜索（交换干员、替换低贡献位）提升总分。
4. 输出 Top-K 方案，并给出每个岗位“替补候选”。

优点：实现快、可解释、便于持续维护。

## 4.4 输出设计
输出建议包含：
- 班次方案表（设施 -> 干员列表）；
- 总收益评分与分项贡献；
- 为什么选这套（触发了哪些联动）；
- 若缺某干员，替代方案是什么。

## 5. 数据来源与同步策略

由于技能来自外部文档，建议建立“抓取 + 人工校验”流程：

1. 初始阶段：手工整理为 `data/operator_skills_v1.json`，确保质量。
2. 增量更新：编写解析脚本定期拉取并比对差异。
3. 发布机制：规则库版本号（例如 `ruleset_version: 2026-03-24`）。
4. 回滚能力：保留历史版本，发现错误可快速回退。

## 7. API 设计草案

- `POST /api/roster/parse`
  - 入参：图片文件（多张）
  - 出参：识别干员列表 + 置信度 + 待确认项

- `POST /api/schedule/generate`
  - 入参：
    - 已拥有干员列表
    - 基建布局参数（252/243、自定义）
    - 策略偏好（稳健/高收益/少换班）
  - 出参：Top-K 排班方案 + 解释信息

- `POST /api/schedule/adjust`
  - 入参：用户手动替换后的方案
  - 出参：更新后的收益评估

## 8. 迭代里程碑（建议）

### M1（1~2 周）：可用原型
- 手动录入干员列表（先跳过 OCR）
- 规则库 V1（覆盖主流制造/贸易组合）
- 能生成基础排班结果

### M2（2~3 周）：接入截图识别
- 图片上传 + OCR + 名称纠错
- 人工确认流程
- 与排班引擎打通

### M3（2 周）：质量提升
- 引入疲劳/轮换简化模型
- 增加可解释性与替代方案
- 建立回归测试集（固定干员池 -> 期望组合）

### M4（持续）：数据与策略迭代
- 更新新干员技能
- 优化启发式权重
- 收集用户反馈做 A/B 对比

## 9. 测试与验收标准

### 9.1 模块 A（识别）
- 干员识别准确率（Top-1）≥ 95%（标准截图集）。
- 低置信度项召回率高（宁可提示确认，不可静默误判）。

### 9.2 模块 B（排班）
- 对基准账号集，方案评分接近人工经验上限（例如达到人工方案 90%+）。
- 输出结果必须可解释（每个岗位给选择理由）。

### 9.3 工程质量
- 单元测试覆盖关键计算逻辑。
- 规则变更触发回归测试。
- 关键接口有示例请求/响应与错误码。

---

## 10. 你可以立刻开工的最小落地步骤

1. 先做 `手动录入干员 -> 排班结果` 的闭环（不依赖 OCR）。
2. 同步整理技能规则库（先覆盖常用高收益组合）。
3. 实现可解释评分器与 Top-K 候选。
4. 再接入 OCR，最后做体验优化与自动更新。

这个路径能最快把“有用的排班建议”做出来，避免在 OCR 上过早卡住。
